# Decision: 모노레포 + 공유 타입 레이어 (libs/)

## Context

프론트엔드(Next.js)와 백엔드(NestJS)가 동일한 도메인 타입을 다룬다.

별도 레포로 분리하면 타입 동기화 문제가 발생한다.

---

## Problem

프론트엔드와 백엔드의 타입을 어떻게 공유할 것인가?

가능한 방향:

1. 각 프로젝트에서 타입을 독립적으로 정의
2. 별도 npm 패키지로 공유 타입 관리
3. 모노레포 내 libs/ 공유 레이어

---

## Decision

모노레포 구조로 프로젝트를 구성하고, `libs/types/`에 공유 도메인 타입을 정의한다.

구조:

```
petlog/
├── frontend/
├── backend/
└── libs/
    ├── types/   ← 공유 도메인 타입
    └── seed/    ← 개발용 시드 데이터
```

---

## Reason

### 타입 단일 출처

도메인 타입을 한 곳에서 관리하면 프론트엔드와 백엔드의 타입 불일치를 방지한다.

### 포트폴리오 관점

실제 서비스에서 모노레포는 일반적인 선택이다.

모노레포 구성 경험 자체가 포트폴리오 가치를 가진다.

### 배포 독립성 유지

libs/는 타입 정의만 포함한다.

런타임 의존성이 없으므로 프론트엔드와 백엔드의 배포는 독립적으로 유지된다.

---

## Trade-off

초기 설정이 단일 레포보다 복잡하다.

1인 개발에서는 과도한 구조일 수 있다.

하지만 타입 불일치로 인한 버그 비용이 더 크다고 판단했다.

---

## Status

현재: libs/types/ 에 도메인 타입 정의 완료
