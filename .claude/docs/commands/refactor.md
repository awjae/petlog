# Refactor 명령어 가이드

## 목적

기존 코드를 리팩토링할 때 따라야 할 원칙과 순서를 제공한다.

리팩토링은 동작을 바꾸지 않고 구조를 개선하는 작업이다.

---

## 1. 리팩토링 전 확인

리팩토링을 시작하기 전에 다음을 확인한다.

- 현재 코드의 동작이 올바른가?
- 어떤 문제를 해결하려는가?
- 범위가 명확한가?

범위가 불명확한 리팩토링은 하지 않는다.

---

## 2. 리팩토링 대상 판단 기준

### 리팩토링이 필요한 신호

- 단일 함수/컴포넌트가 여러 책임을 가지고 있다
- 비즈니스 로직이 Controller 또는 Component에 있다
- 중복 코드가 3곳 이상 존재한다
- 타입이 `any`로 처리되어 있다
- 데이터 흐름이 단방향을 벗어나 있다

### 리팩토링 하지 않아도 되는 경우

- 단순히 스타일이 마음에 들지 않는 경우
- 동작에 문제가 없고 읽기 어렵지 않은 경우
- 범위가 너무 커서 기능 변경을 동반하는 경우

---

## 3. 리팩토링 원칙

### 동작 보존

리팩토링 전후 외부 동작이 동일해야 한다.

기능 추가와 리팩토링을 동시에 하지 않는다.

### 작은 단위

한 번에 하나의 책임만 분리한다.

큰 리팩토링은 단계적으로 진행한다.

### 아키텍처 방향 유지

리팩토링 후에도 다음 원칙을 지킨다.

- Domain First
- Separation of Concerns
- 단방향 데이터 흐름

---

## 4. 계층별 리팩토링 기준

### Frontend

Component 리팩토링:

- UI 렌더링만 담당하도록 분리
- 비즈니스 로직은 Hook으로 이동
- API 호출은 API Layer로 이동

Hook 리팩토링:

- 하나의 Hook이 하나의 책임만 가지도록 분리
- 데이터 가공 로직은 util 함수로 분리

### Backend

Controller 리팩토링:

- Request / Response 처리만 남긴다
- 비즈니스 로직은 Service로 이동

Service 리팩토링:

- 도메인 로직이 Service에 모이도록 정리
- 외부 서비스 호출은 별도 Provider로 분리

---

## 5. 완료 체크리스트

- 동작이 변경되지 않았는가?
- 책임이 명확하게 분리되었는가?
- 데이터 흐름이 단방향을 유지하는가?
- `any` 타입이 제거되었는가?
- 기존 기능이 정상 동작하는가?
