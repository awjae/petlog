# Review 명령어 가이드

## 목적

코드 리뷰 시 Petlog 아키텍처 원칙을 기준으로 일관되게 검토할 수 있도록 체크리스트를 제공한다.

---

## 1. 리뷰 관점

코드 리뷰는 다음 네 가지 관점으로 진행한다.

### 제품 관점

- 사용자 문제를 해결하는 변경인가?
- MVP 범위에 맞는가?
- 사용자 경험을 해치지 않는가?

### 아키텍처 관점

- Domain 경계를 지키는가?
- 책임이 명확하게 분리되어 있는가?
- 데이터 흐름이 단방향인가?

### 코드 품질 관점

- 읽기 쉬운가?
- 변수명과 함수명이 의도를 드러내는가?
- 불필요한 복잡성이 없는가?

### 안정성 관점

- Error Case가 처리되어 있는가?
- 타입이 명확하게 정의되어 있는가?
- 엣지 케이스를 고려했는가?

---

## 2. Frontend 리뷰 체크리스트

### Component

- Component가 UI 렌더링에만 집중하는가?
- 비즈니스 로직이 Component에 있지 않은가?
- API 호출을 Component에서 직접 하지 않는가?
- Props 타입이 명확하게 정의되어 있는가?

### Hook

- Hook이 단일 책임을 가지는가?
- 데이터 흐름이 Component → Hook → API Layer 순서인가?

### 상태 관리

- Local State / Server State 구분이 명확한가?
- 불필요한 전역 상태가 없는가?

### UX

- Loading 상태가 처리되어 있는가?
- Empty 상태가 처리되어 있는가?
- Error 상태가 처리되어 있는가?
- 모바일 레이아웃에서 동작하는가?

---

## 3. Backend 리뷰 체크리스트

### Controller

- Controller가 Request / Response 처리만 담당하는가?
- 비즈니스 로직이 Controller에 없는가?
- DTO가 명확하게 정의되어 있는가?

### Service

- 도메인 로직이 Service에 있는가?
- 외부 서비스 의존성이 추상화되어 있는가?
- AI 호출이 인터페이스를 통해 이루어지는가?

### 권한 검증

- 다른 사용자의 Pet에 접근하지 못하도록 검증하는가?
- 인증이 필요한 API에 Guard가 적용되어 있는가?

---

## 4. TypeScript 리뷰 체크리스트

- `any` 타입이 없는가?
- Domain Type이 명확하게 정의되어 있는가?
- 타입 오류를 숨기는 캐스팅이 없는가?

---

## 5. 최종 판단 기준

변경을 수락하기 위한 기준:

1. 사용자 가치가 있는가?
2. 아키텍처 원칙을 따르는가?
3. 읽기 쉽고 유지보수 가능한가?
4. 안전하게 동작하는가?

위 기준 중 하나라도 만족하지 않으면 개선을 요청한다.
