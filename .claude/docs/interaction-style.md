# Claude 상호작용 스타일 가이드

## 꼬리질문 규칙

작업 완료 후 또는 계획 수립 후에는 반드시 아래 형식으로 꼬리질문 3개를 제시한다.

---

## 언제 적용하는가

### 적용하는 상황 (큰 작업)

다음 중 하나 이상에 해당할 때 꼬리질문을 제시한다.

- 새로운 기능 또는 모듈을 구현했을 때
- 아키텍처 또는 설계 계획을 제안했을 때
- 여러 파일에 걸친 리팩토링을 마쳤을 때
- 데이터 모델 또는 API 구조를 변경했을 때

### 적용하지 않는 상황 (작은 작업)

형식적인 꼬리질문이 노이즈가 되는 상황은 생략한다.

- 오타 수정, 변수명 변경, 주석 수정 등 단순 수정
- 단일 파일의 사소한 스타일 변경
- 질문에 대한 설명 또는 코드 해석만 했을 때

판단 기준: 이 변경이 설계 결정을 포함하는가? 포함하면 적용, 아니면 생략.

---

## 질문 형식

작업 완료 후 마지막에 다음 형식으로 제시한다.

```
---

**추가로 고려할 점이 있습니다.**

1. [우려 또는 고려사항 질문]
2. [우려 또는 고려사항 질문]
3. [우려 또는 고려사항 질문]
```

---

## 질문 기준

질문은 다음 관점에서 선택한다.

### 아키텍처 관점

- 이 변경이 기존 구조와 충돌하지 않는가?
- 도메인 경계를 지키고 있는가?

### 확장성 관점

- 향후 기능 추가 시 이 구조가 걸림돌이 되지 않는가?
- AI Provider 교체 또는 다른 확장 시나리오에서 유연한가?

### 제품 관점

- 사용자 경험에 부정적인 영향이 없는가?
- Edge Case나 에러 상황을 충분히 고려했는가?

### 운영 관점

- 배포 또는 마이그레이션 시 리스크가 있는가?
- 성능 또는 데이터 정합성 문제가 생길 수 있는가?

---

## 질문 품질 기준

좋은 꼬리질문:

- 현재 작업에서 놓쳤을 수 있는 구체적인 문제를 지목한다
- "네 / 아니오"로 끝나지 않고 판단을 요구한다
- 이미 처리된 내용은 다시 묻지 않는다

나쁜 꼬리질문:

- "잘 동작하는지 확인하셨나요?" (막연함)
- "테스트는 필요하지 않으신가요?" (반복적)
- "추가 기능이 필요하신가요?" (범위 무관)
